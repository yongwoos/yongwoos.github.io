# 2. 문제 상황 (왜 필요했냐)

운영체제 입장에서는 I/O 대상이 너무 많다:

* 디스크 파일
* 소켓 (TCP/UDP)
* 파이프
* 디바이스 (키보드, 터미널)

👉 각각 접근 방식이 다르면?

* API 다 다름 ❌
* 코드 복잡도 폭발 ❌

---

# 3. 해결 아이디어

👉 **“전부 파일처럼 취급하자”**

그래서 나온 개념:

👉 **Everything is a file**

---

# 4. fd (file descriptor)의 역할

```c
int fd = open("file.txt", ...);
read(fd, buf, size);
write(fd, buf, size);
```

👉 여기서 중요한 건:

👉 **fd는 “I/O 객체를 가리키는 공통 핸들”**

---

# 5. 내부 구조 (핵심)

## 유저 → 커널 흐름

```id="3zv7y9"
프로세스
 ↓
fd 테이블 (배열)
 ↓
file 객체
 ↓
inode / socket / pipe
 ↓
실제 디바이스 or 네트워크
```

---

## 단계별 설명

### ① fd 테이블 (프로세스별)

* `fd`는 그냥 인덱스
* 각 프로세스마다 따로 존재

---

### ② file 객체 (커널)

👉 진짜 핵심

* 현재 offset
* 접근 모드
* 파일 연산 함수 포인터

---

### ③ inode / socket 등

* 디스크 파일 → inode
* 네트워크 → socket 구조체
* 파이프 → pipe 구조체

👉 타입은 다르지만…

---

## 6. VFS의 핵심 역할

👉 **다른 파일 시스템/객체를 하나로 통일**

예:

* ext4
* xfs
* nfs
* socket
* pipe

👉 전부 공통 인터페이스 제공:

* open
* read
* write
* close

---

## 7. 왜 가능한가 (진짜 핵심)

👉 **함수 포인터 테이블 (ops 구조)**

각 객체는 이런 걸 가짐:

* file_operations

예:

* 디스크 파일 → disk read/write 함수
* 소켓 → network send/recv 함수

👉 그런데 호출은 동일:

```c
read(fd, ...)
```

→ 내부에서 타입에 맞는 함수 실행됨

---

## 8. 그래서 생기는 엄청난 장점

### ① 코드 단순화

```c
read(fd, ...)
```

→ 파일이든 소켓이든 동일

---

### ② 확장성

* 새로운 파일 시스템 추가 가능
* 네트워크도 파일처럼 처리

---

### ③ 재사용성

* `select`, `epoll` 같은 것도 fd 기반

---

## 9. 소켓도 fd인 이유

👉 소켓도 결국:

* file 객체 하나 생성
* file_operations가 “네트워크용”

그래서:

```c
read(sock, ...)
write(sock, ...)
```

👉 파일이랑 똑같이 동작

---

## 10. 한 줄 핵심

👉 **fd는 “모든 I/O를 하나로 다루기 위한 공통 인터페이스”이고, VFS가 그걸 가능하게 한다**

---

## 최종 정리

* fd = 커널 객체를 가리키는 핸들
* VFS = 다양한 I/O를 하나로 통합
* 내부는 구조체 + 함수 포인터로 분기
* 파일, 소켓, 파이프 전부 동일 API 사용
