title: 세그먼트/바이트스트림
---
👉 **TCP는 “연속된 바이트 스트림이 끊기지 않는 지점까지” 모아서 응용계층으로 넘긴다.**

TCP는 데이터를 **세그먼트 단위로 받지만, 다시 이어붙여서 바이트 스트림으로 응용에 전달**

## 1. 동작 핵심 구조

TCP는 패킷(세그먼트) 단위가 아니라
👉 **바이트 스트림(sequence number 기반)** 으로 동작한다.

각 세그먼트는 이런 느낌:

* seq = 1000 (1000~1499 바이트)
* seq = 1500 (1500~1999 바이트)
* seq = 2000 ...

---

## 2. 언제 응용계층으로 전달하냐

조건은 딱 하나:

### ✔ **중간에 빈 구간 없이 “연속적으로 이어졌을 때”**

예시 보자

### 정상 케이스

```
[1000~1499] 도착
[1500~1999] 도착
[2000~2499] 도착
```

👉 1000부터 2499까지 연속

→ **한 번에 응용계층으로 전달 가능**

---

### out-of-order 케이스

```
[1000~1499] 도착
[2000~2499] 도착  (중간 빠짐)
```

👉 1500~1999 없음 (hole 발생)

→ TCP는:

* 2000번대 데이터 **버퍼에 저장만 하고**
* **응용계층에 절대 안 넘김**

---

### 이후 복구

```
[1500~1999] 도착
```

👉 이제 1000~2499 연속 완성

→ **그 순간 한 번에 응용계층으로 전달**

---

## 3. 정리 (시험용 핵심)

* TCP는 **순서 보장**
* **out-of-order 데이터는 버퍼링**
* **hole(빈 구간)이 있으면 전달 안함**
* **연속된 구간이 완성되는 순간 전달**

---

## 4. 추가로 중요한 포인트 (실무 감각)

응용계층에서 `read()` 할 때:

* TCP는 “세그먼트 단위”로 주는 게 아님
* 그냥 **현재까지 연속된 바이트 덩어리** 줌

즉:

* 1번 read → 1000바이트
* 2번 read → 300바이트

👉 **경계 없음 (message boundary 없음)**


## 한 줄 결론

👉 TCP는
**“중간에 끊김 없이 이어진 바이트 구간이 생길 때까지 기다렸다가 그때 전달한다.”**


# TCP 메시지의 끝?
TCP는 **“끝”** 이라는 개념이 없습니다. TCP는 단지 **“연속된 바이트 스트림이 끊기지 않는 지점까지”** 모아서 응용계층으로 넘길 뿐입니다.

👉 **TCP는 “끝”을 모른다.**
👉 **그냥 도착한 만큼 계속 올려준다.**


## 1. TCP는 “메시지 개념 자체가 없음”

TCP는 이런 식으로 생각해야 된다:

* ❌ “한 덩어리 다 모이면 전달”
* ❌ “끝까지 다 받으면 전달”

👉 이런 개념 **없음**

대신:

👉 **그냥 바이트 흐름(stream)**
👉 **연속된 만큼 계속 응용계층으로 밀어줌**

---

## 2. “언제까지 모으냐?” → 답

👉 **모으는 게 아니라, 연속된 부분이 생길 때마다 바로 전달**

예시

### 상황

* 1000~1499 도착
* 1500~1999 도착

👉 이 시점에서 이미 연속

→ **바로 응용계층으로 전달 가능**

굳이 더 기다리지 않는다.

---

## 3. “끝”은 누가 아냐?

👉 **응용계층이 안다. TCP는 모른다.**

방법은 3가지다:

---

### ① 길이 기반 (Length field)

예: HTTP Content-Length

* “앞에서 1000바이트 온다”
* 응용이 1000바이트 다 읽으면 끝

---

### ② 구분자 기반 (Delimiter)

예: 줄바꿈 `\n`

* `hello\nworld\n`
* 응용이 `\n` 기준으로 끊음

---

### ③ 연결 종료 (FIN)

* 송신자가 FIN 보내면
  👉 “이제 데이터 끝이다”

---

## 4. 왜 이렇게 설계됐냐

TCP 역할은 딱 이것뿐이다:

👉 **순서 맞추고, 유실 없이 전달**

끝 판단까지 하면:

* 유연성 깨짐
* 프로토콜마다 요구 다름 (HTTP, FTP, DB 등)

그래서:

👉 **“끝 판단은 응용계층 책임”**

---

## 5. 한 방에 정리

* TCP는 세그먼트 단위 ❌ → 바이트 스트림 ⭕
* “언제까지 모음?” ❌ → 그런 개념 없음
* **연속된 바이트 생기면 바로 전달**
* “끝 어디냐?” → **응용이 정의**

---

## 한 줄 핵심

👉 TCP는
**“끝을 기다리는 게 아니라, 이어진 만큼 계속 흘려보내는 파이프”다**

# 바이트스트림이 세그먼트인 이유
👉 **TCP는 “논리적으로는 바이트 스트림”**
👉 **물리적으로는 “세그먼트 단위로 쪼개서 보냄”**


## 1. 왜 바이트 스트림이냐 (논리)

TCP 입장에서 데이터는 이렇게 본다:

```
[0][1][2][3][4][5]...
```

👉 그냥 **끊김 없는 바이트 흐름**

* 메시지 경계 없음
* “이게 한 덩어리” 이런 개념 없음
* 순서만 중요 (sequence number)


## 2. 근데 왜 세그먼트냐 (현실)

네트워크로 보낼 때는:

👉 한 번에 못 보냄
👉 **적당한 크기로 잘라야 함**

이게 바로:

👉 **세그먼트 (segment)**


### 예시

응용이 5000바이트 보냄

TCP 내부:

```
→ 1460
→ 1460
→ 1460
→ 620
```

👉 이렇게 쪼개서 전송

각각이 **TCP 세그먼트**


## 3. 핵심 차이 (시험 핵심)

| 구분      | 의미         |
| ------- | ---------- |
| 바이트 스트림 | 논리적 데이터 모델 |
| 세그먼트    | 전송 단위      |

---

## 4. 중요한 오해 깨기

❌ “세그먼트 = 메시지 단위”
→ 완전 틀림

❌ “세그먼트 단위로 응용에 전달”
→ 아님

👉 TCP는:

* 세그먼트로 받지만
* **다시 이어붙여서**
* **바이트 스트림으로 응용에 전달**

## 5. 한 방 비유

👉 수도관 생각하면 된다

* 물 흐름 = 바이트 스트림
* 물통으로 나눠 운반 = 세그먼트

사용자는:

👉 물통 단위 신경 안 씀
👉 그냥 물 나오는 것만 봄

---

## 한 줄 핵심

👉 **TCP는 “쪼개서 보내지만, 이어서 전달한다”**
