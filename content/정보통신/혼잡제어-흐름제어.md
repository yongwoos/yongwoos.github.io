# 1. 슬라이딩 윈도우 (Sliding Window)

👉 한 줄 정의

👉 **“ACK 안 받고도 미리 보내도 되는 데이터 범위”**

---

## 구조 감각

```
[보낸거][보냈지만 ACK 안옴][아직 못보냄]
         ↑
     이게 윈도우
```

---

## 핵심 동작

* TCP는 한 번 보내고 멈추지 않음
* 일정 범위까지 **연속으로 전송**
* ACK 오면 → 윈도우 앞으로 이동 (슬라이딩)

---

## 왜 필요하냐

👉 RTT 기다리면 너무 느림

그래서:

👉 **“파이프라인처럼 미리 밀어 넣는 구조”**

---

## 핵심 포인트

* 윈도우 크기 = 성능
* 크면 → 빠름
* 작으면 → 느림

---

# 2. Flow Control (흐름 제어)

👉 한 줄 정의

👉 **“수신자가 감당 가능한 속도로 보내라”**

---

## 어떻게 구현됨?

👉 recv buffer 기반

수신자가 이렇게 말함:

```
"나 지금 3000바이트 받을 수 있음"
```

→ 이게 **윈도우 크기 (rwnd)**

---

## 상황

### recv buffer 꽉 참

👉 수신자:

```
"0 바이트 가능"
```

→ 송신자: **전송 멈춤**

---

## 핵심

👉 **속도 기준은 ‘네트워크’가 아니라 ‘수신자 처리 속도’**

---

# 3. Congestion Control (혼잡 제어)

👉 한 줄 정의

👉 **“네트워크 터지지 않게 천천히 보내라”**

---

## Flow Control vs Congestion Control

| 구분                 | 기준      |
| ------------------ | ------- |
| Flow Control       | 상대 컴퓨터  |
| Congestion Control | 네트워크 상태 |

---

## 핵심 변수

👉 **cwnd (congestion window)**

실제 전송 가능량:

$$
\text{실제 윈도우} = \min(\text{rwnd}, \text{cwnd})
$$

---

## 동작 요약

### 처음

👉 천천히 시작 (slow start)

### 문제 없음

👉 점점 증가

### 패킷 유실 발생

👉

* 속도 확 줄임
* 다시 천천히 증가

---

## 감각

👉 TCP는 이렇게 생각함:

* “지금 괜찮네?” → 속도 ↑
* “어? 터졌다” → 속도 ↓↓↓

---

# 4. 왜 recv buffer 꽉 차면 느려지냐

이건 진짜 중요하다.

---

## 흐름

1. recv buffer 꽉 참
2. 수신자:

```
rwnd = 0
```

3. 송신자:

👉 **전송 중단**

---

## 결과

👉 네트워크 문제가 아니라:

👉 **애플리케이션이 데이터를 안 읽어서 막힘**

---

## 핵심 한 줄

👉 **“수신 앱이 read() 안 하면 TCP 전체가 멈춘다”**

---

# 5. 전체 그림 (진짜 핵심)

TCP 속도는 이 2개가 결정함:

$$
\text{전송량} = \min(\text{수신 버퍼}, \text{네트워크 상태})
$$

---

# 최종 한 줄 정리

👉 **슬라이딩 윈도우 = 미리 보내는 범위**
👉 **Flow Control = 상대가 감당 가능한 속도 제한**
👉 **Congestion Control = 네트워크 터지지 않게 조절**
