---
title: HDLC
---
HDLC(High-Level Data Link Control)는 **데이터 링크 계층(OSI 2계층)**에서 사용되는 **비트 지향형 통신 프로토콜**입니다. 주로 장치 간 신뢰성 있는 데이터 전송을 위해 설계되었습니다.

---

## 🔹 1. HDLC 개요

* ISO에서 표준화한 프로토콜
* 동기식(serial) 통신에서 사용
* 오류 검출 및 흐름 제어 기능 제공
* WAN(광역 네트워크) 환경에서 많이 사용됨
* 단방향, 양방향 모두 지원
* 오류 제어로 ARQ 사용

---

## 🔹 2. 특징

* **비트 지향 프로토콜** → 문자 단위가 아니라 비트 단위 처리
* **프레임 기반 통신**
* **오류 검출**: CRC 사용
* **흐름 제어 및 순서 제어**
* **다양한 구성 지원**

  * Point-to-Point
  * Multipoint


## 🔹 3. 프레임 구조

HDLC는 다음과 같은 프레임 형식을 가짐:

```
Flag | Address | Control | Data | FCS | Flag
```

### 각 필드 설명

* **Flag (01111110)**
  → 프레임 시작/끝 표시

* **Address**
  → 송수신 장치 주소

* **Control**
  → 프레임 종류 및 제어 정보

* **Data**
  → 실제 데이터

* **FCS (Frame Check Sequence)**
  → 오류 검출 (CRC)

### 비트 스터핑(Bit Stuffing)의 목적
플래그 동기 방식(HDLC 등)에서 프레임의 시작과 끝을 알리는 플래그 패턴($01111110$)이 실제 데이터 안에 포함되어 있을 때 발생하는 문제를 해결하기 위한 기술입니다

* **데이터 투명성(Transparency)**: 사용자 데이터 내에 플래그와 동일한 패턴($1$이 6개)이 나타나면, 수신 측은 이를 프레임의 끝으로 오해할 수 있습니다.

* **해결 방법**: 송신 측에서 데이터 중 $1$이 연속으로 5개 나오면 무조건 뒤에 $0$을 강제로 삽입(Stuffing)합니다.

* **수신 측 처리**: 수신 측은 $1$이 5개 연속된 후 나오는 $0$을 제거하여 원래 데이터를 복원합니다. 만약 $1$이 6개라면 그것을 진짜 플래그로 인식합니다.


## 🔹 4. 프레임 종류

HDLC는 3가지 프레임 타입을 사용:

### ① I-Frame (Information)

* 데이터 전달용
* 순서 번호 포함

### ② S-Frame (Supervisory)

* 흐름 제어 및 오류 제어
* ACK/NACK 역할

### ③ U-Frame (Unnumbered)

* 연결 설정 및 해제 등 제어용

## 🔹 5. 동작 모드

HDLC는 3가지 모드 지원:

* **NRM (Normal Response Mode)**

  * 주국(primary) → 종국(secondary) 제어
* **ARM (Asynchronous Response Mode)**

  * 종국이 자율적으로 응답 가능
* **ABM (Asynchronous Balanced Mode)**

  * 양쪽이 동등 (가장 많이 사용)

## 🔹 6. 장점 / 단점

### ✔ 장점

* 높은 신뢰성
* 다양한 네트워크 구성 지원
* 효율적인 오류 처리

### ✖ 단점

* 구현이 복잡
* 오버헤드 존재


### 핵심 요약
| 기술 명칭 | 적용 방식 | 주요 목적 |
| :--- | :--- | :--- |
| **비트 스터핑** | 1이 5개 연속되면 0 삽입 | **데이터 투명성 보장** (데이터와 플래그 구분) |
| **바이트 스터핑** | 특수 문자(ESC) 삽입 | 문자 위주 프레임에서 제어 문자와 데이터 구분 |

## 🔹 7. 한 줄 정리

👉 **HDLC = 비트 단위 프레임 기반, 오류 제어와 흐름 제어를 제공하는 데이터 링크 계층 프로토콜**
