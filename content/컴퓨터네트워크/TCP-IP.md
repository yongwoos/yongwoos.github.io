# TCP의 마법: 조각난 패킷을 어떻게 완벽하게 재조립할까?

네트워크상에서 데이터는 한꺼번에 흐르지 않고 **패킷(Packet)** 이라는 작은 단위로 쪼개져서 전달됩니다. 이때 각 패킷은 서로 다른 경로로 전달될 수 있어, 도착 순서가 뒤바뀌는 **역전 현상(Out-of-order)** 이 발생하곤 합니다.

## 1. 시퀀스 번호(Sequence Number)의 역할
TCP 헤더에는 각 바이트마다 부여되는 **시퀀스 번호**가 있습니다.
* **송신측:** 데이터의 몇 번째 바이트인지 번호를 매겨서 보냅니다.
* **수신측:** 이 번호를 보고 "아, 이게 1번 조각 다음인 2번 조각이구나"를 판단합니다.

## 2. 수신 버퍼와 재조립 과정
수신측에는 데이터를 임시로 담아두는 **수신 버퍼(Receive Buffer)** 가 있습니다.
1.  **도착:** 패킷이 들어오면 일단 시퀀스 번호를 확인합니다.
2.  **정렬:** 1번, 3번이 먼저 도착했다면, 3번을 버퍼의 뒤쪽 칸에 일단 넣어두고 2번이 올 때까지 기다립니다.
3.  **재조립:** 드디어 2번이 도착하면 비어있던 칸을 채워 1-2-3 순서를 완성합니다.
4.  **전달:** 순서가 완벽하게 맞춰진 데이터만 위쪽의 응용 프로그램(HTTP, FTP 등)으로 올려보냅니다.



## 3. 슬라이딩 윈도우와의 관계
앞서 배운 **슬라이딩 윈도우**는 이 재조립 과정을 효율적으로 만듭니다.
* **윈도우 크기:** 수신측이 "지금 내 버퍼에 빈 자리가 이만큼 있으니, 순서가 좀 바뀌더라도 이 범위 내에서는 계속 보내도 돼"라고 송신측에 알려주는 값입니다.
* **흐름 제어:** 만약 재조립이 늦어져 버퍼가 가득 차면, 윈도우 크기를 0으로 만들어 송신측을 잠시 멈추게 함으로써 데이터 유실을 막습니다.

## 4. 에러 복구 (ARQ와의 만남)
만약 특정 조각(예: 2번)이 끝까지 오지 않는다면?
* **수신측은 3번, 4번을 다 받아놓고도 **"2번이 비었어!"** 라며 ACK를 통해 재전송을 요구합니다.
* 이때 **Selective Repeat(선택적 재전송)** 방식을 지원한다면, 이미 잘 받은 3, 4번은 놔두고 2번만 쏙 골라서 다시 받게 됩니다.


## 💡 요약
TCP는 **"다 받아놓고 조립하는 것"** 이 맞지만, 내부적으로는 **"먼저 온 조각들을 번호순으로 빈칸에 배치하며 기다리다가, 이 빠진 조각이 채워지는 즉시 순서대로 합쳐서 위로 올리는"** 아주 역동적인 과정을 거칩니다.

클라우드 엔지니어로 일하시다 보면 대용량 트래픽 처리 시 `TCP Window Full`이나 `Out-of-Order` 로그를 보실 때가 있을 텐데, 바로 이 재조립 과정에서 병목이 생기는 현상이라고 이해하시면 완벽합니다! 

# ⏱️ TCP ACK는 도대체 언제 보낼까?
TCP의 ACK(확인 응답) 송신 타이밍은 **"데이터를 다 받아서 조립이 끝난 후"가 아닙니다.** 그렇게 하면 네트워크 속도가 너무 느려지기 때문이죠.

TCP는 데이터가 조립되기를 기다리지 않고, **패킷을 수신하는 즉시(혹은 아주 짧은 지연 후)** ACK를 보냅니다.

많은 분이 "데이터를 다 받고 조립이 끝나야 ACK를 보낸다"고 오해하지만, 실제 TCP는 훨씬 더 부지런하게 움직입니다.

## 1. 패킷 수신 즉시 (세그먼트 단위)
수신 측은 패킷(세그먼트)이 도착하면 해당 데이터를 잘 받았다는 것을 알리기 위해 즉시 ACK를 생성합니다. 이때 ACK 번호는 **"다음에 받기를 기대하는 바이트 번호"** 를 의미합니다.
* 예: 1~100번 데이터를 받으면, ACK 101을 보냄 ("100번까지 잘 받았으니 이제 101번 줘!")

## 2. 지연된 응답 (Delayed ACK)
매 패킷마다 ACK를 보내면 네트워크 부하가 커지기 때문에, 효율을 위해 **약간 기다렸다가** 보냅니다.
* **동작:** 패킷 하나를 받으면 아주 잠시(보통 200ms 미만) 기다립니다.
* **효과:** 그 사이 두 번째 패킷이 오면 ACK 하나로 두 패킷을 동시에 확인해줄 수 있고, 마침 상대방에게 보낼 데이터가 있다면 **피기백(Piggyback)** 응답에 실어 보낼 수도 있습니다.



## 3. 순서가 어긋난 경우 (Out-of-Order) → 즉시 발송!
이 부분이 중요합니다. 만약 1번 다음에 3번이 먼저 도착했다면?
* 수신 측은 조립이 안 된 상태여도 **즉시** ACK를 보냅니다.
* 하지만 이때 ACK 번호는 4번이 아니라 **여전히 2번**입니다. ("난 아직 2번이 필요해!")
* 이를 통해 송신 측은 "아, 2번이 중간에 유실됐구나"라고 빠르게 눈치챌 수 있습니다. (**빠른 재전송** 유도)



## 4. 조립과 ACK의 차이점
* **ACK 송신:** 물리적으로 패킷이 들어와서 **버퍼에 안전하게 저장**되는 순간 발생합니다. (네트워크 계층 간의 약속)
* **재조립:** 버퍼에 쌓인 조각들의 순서가 다 맞아서 **응용 프로그램(HTTP 등)이 읽어갈 수 있게** 합쳐지는 과정입니다. (애플리케이션 계층으로 넘겨주는 단계)

---

## 💡 결론: 엔지니어링 포인트
ACK는 **"나 이만큼은 확실히 내 주머니(버퍼)에 챙겼어"**라는 영수증과 같습니다. 조립은 주머니에 챙긴 것들을 꺼내서 나중에 순서대로 맞추는 별개의 작업입니다.

이 **ACK 타이밍**이 꼬이면 네트워크 지연 시간(Latency)이 늘어납니다. 클라우드 환경에서 `TCP 3-way handshake` 이후 데이터 전송 속도가 안 나온다면, 이 ACK 응답이 지연되고 있지는 않은지 확인해봐야 하죠.

**요약:** 패킷이 들어올 때마다(혹은 두 개당 하나씩) 바로바로 보냅니다. 순서대로 다 모일 때까지 기다리는 건 **애플리케이션**이지 **TCP**가 아닙니다! 이해가 좀 더 명확해지셨나요?

블로그에 용어 정리용으로 올리기 좋게 **세그먼트(Segment)**와 **패킷(Packet)**의 차이점을 계층별 구조와 함께 정리해 드릴게요.

결론부터 말하면, 이 둘은 **"데이터가 어느 계층(Layer)에 있느냐"**에 따라 부르는 이름이 다른 것뿐입니다.


# 📦 세그먼트(Segment) vs 패킷(Packet): 데이터의 이름표

네트워크에서 전송되는 데이터 단위는 각 계층을 통과할 때마다 특정한 정보를 담은 **헤더(Header)**가 붙는데, 이를 **PDU(Protocol Data Unit)**라고 부릅니다. 이 PDU가 계층마다 이름이 달라집니다.

## 1. 세그먼트 (Segment)
* **계층:** 제4계층 - **전송 계층(Transport Layer)**
* **주요 프로토콜:** TCP
* **특징:** 상위 계층에서 내려온 방대한 데이터를 네트워크로 보내기 적당한 크기로 쪼갠 상태입니다.
    * 헤더에 **포트 번호(Port Number)**와 **시퀀스 번호(Sequence Number)**가 포함됩니다.
    * 송신측과 수신측의 **프로세스 간 연결**을 관리하는 단계에서의 이름입니다.

## 2. 패킷 (Packet)
* **계층:** 제3계층 - **네트워크 계층(Network Layer)**
* **주요 프로토콜:** IP
* **특징:**
    * 세그먼트에 **IP 주소(출발지/목적지 IP)**가 적힌 헤더가 붙은 상태입니다.
    * 데이터가 실제 경로를 찾아 떠날 수 있는 **최소한의 기본 단위**가 됩니다.
    * 라우터(Router)가 이 패킷의 IP 정보를 보고 경로를 설정(Routing)합니다.



---

## 3. 한눈에 비교하기

| 구분 | 세그먼트 (Segment) | 패킷 (Packet) |
| :--- | :--- | :--- |
| **소속 계층** | 4계층 (Transport) | 3계층 (Network) |
| **핵심 정보** | 포트 번호, 순서 제어 | **IP 주소**, 경로 제어 |
| **담당 장비** | 게이트웨이, L4 스위치 | **라우터(Router)** |
| **비유** | 편지지에 적힌 **내용의 순서** | 주소가 적힌 **편지 봉투** |

---

## 4. 참고: 다른 이름들
데이터가 더 내려가거나 올라가면 또 이름이 바뀝니다.
* **프레임(Frame):** 2계층(데이터 링크)에서 물리적 주소인 **MAC 주소**가 붙었을 때의 이름입니다.
* **비트(Bit):** 1계층(물리 계층)에서 전기적 신호(0과 1)로 변환된 상태입니다.

## 💡 엔지니어의 한마디
클라우드 인프라를 다룰 때 사용하는 **MTU(Maximum Transmission Unit)** 설정은 대개 3계층인 **패킷**의 최대 크기를 결정하는 설정입니다. 이 패킷 안에 담기는 순수 데이터의 조각이 **세그먼트**인 셈이죠. 블로그에 정리하실 때 이 계층 구조도(Encapsulation)를 함께 넣으시면 완벽한 설명이 됩니다!

# TCP/IP는 프로그램인가?
짧게 결론부터:

👉 **TCP/IP는 “프로그램도, 단일 객체도 아니다”**
👉 **OS 커널 안에 구현된 “프로토콜 스택(코드 + 자료구조 집합)”이다**

---

## 1. 왜 프로그램이 아니냐

* 하나의 실행 파일처럼 따로 돌아가는 프로세스 ❌
* 네가 `ps`로 볼 수 있는 대상도 아님

👉 대신:

👉 **커널 내부 코드가 이벤트 발생 시 실행됨**

예:

```c
write(sock, ...)
```

→ 커널 진입
→ TCP 코드 실행
→ 세그먼트 생성/전송

---

## 2. 왜 객체도 아니냐

* “TCP 하나”라는 단일 객체 ❌

대신:

👉 **연결마다 상태 객체가 따로 있음**

예:

* 소켓 구조체 (`struct sock`)
* 시퀀스 번호
* 윈도우 크기
* 재전송 큐

👉 연결 수만큼 여러 개 존재

---

## 3. 정확한 정체

👉 TCP/IP =

* 프로토콜 구현 코드 (로직)
* * 상태를 담는 구조체들
* * 버퍼
* * 타이머

👉 즉:

👉 **“동작하는 코드 + 상태 객체들의 집합”**

---

## 4. 어떻게 동작하냐 (이벤트 기반)

### 보내기

```c
write(sock, ...)
```

→ 커널
→ TCP send 로직 실행
→ 세그먼트 생성
→ IP로 전달

---

### 받기

→ NIC 인터럽트 발생
→ 커널 진입
→ IP → TCP 처리
→ 소켓 버퍼에 저장

---

## 5. 소켓이랑 관계

* 소켓 = “통신 창구 (객체)”
* TCP = “그 창구 뒤에서 실제 일하는 엔진”

👉 관계:

```id="g0ow0v"
응용
 ↓
소켓 (구조체)
 ↓
TCP/IP 스택 (코드 + 상태)
 ↓
네트워크
```

---

## 6. 한 줄 정리

👉 **TCP/IP는 “실행 프로그램”이 아니라, 커널 안에서 동작하는 네트워크 처리 로직이다**
