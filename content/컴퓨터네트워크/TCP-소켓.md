## 1. 소켓이란?

👉 **네트워크 통신을 위한 파일 디스크립터/구조체 (인터페이스)**

- OS가 제공하는 통신 창구
- 파일처럼 `read/write` 가능

---

## 2. 전체 구조

```
[응용 프로그램]
↓
소켓
↓
TCP
↓
IP
↓
네트워크
```
## 3. TCP 서버/클라이언트 흐름

### 3.1 서버

```c
socket()
bind()
listen()
accept()
read()/write()
close()
````

### 단계 설명

1. `socket()`

   * 소켓 생성

2. `bind()`

   * IP + PORT 지정

3. `listen()`

   * 연결 대기 상태

4. `accept()`

   * 클라이언트 연결 수락
   * 👉 여기서 **새로운 소켓 생성됨 (중요)**

5. `read()/write()`

   * 데이터 송수신

---

## 3.2 클라이언트

```c
socket()
connect()
read()/write()
close()
```

---

## 4. 중요한 포인트 (시험 + 실무 핵심)

### 4.1 accept()의 의미

👉 서버는 소켓 2개를 쓴다

| 종류               | 역할    |
| ---------------- | ----- |
| listening socket | 연결 받기 |
| connected socket | 실제 통신 |

👉 `accept()` 하면 **새 소켓 생성**

---

### 4.2 TCP는 바이트 스트림

👉 메시지 단위 없음

문제 상황:

```c
send("hello");
send("world");
```

클라이언트:

```c
recv() → "helloworld"
또는
recv() → "hel"
recv() → "loworld"
```

👉 경계 없음

---

### 4.3 그래서 생기는 문제 (Sticky Packet)

👉 데이터 경계가 붙거나 쪼개짐

해결 방법:

* 길이 헤더 붙이기
* 구분자 사용 (`\n`)
* 고정 길이

---

### 4.4 부분 송수신 (핵심)

#### send()

👉 한 번에 다 안 보내질 수 있음

```c
send(sock, buf, len, 0);
```

→ 실제로는 일부만 전송 가능

---

#### recv()

👉 요청한 길이보다 적게 올 수 있음

```c
recv(sock, buf, 1000, 0);
```

→ 300바이트만 올 수도 있음

---

👉 반드시 루프 필요

---

## 5. 간단 예제 (C)

### 서버

```c
int server_fd = socket(AF_INET, SOCK_STREAM, 0);

bind(server_fd, ...);
listen(server_fd, 5);

int client_fd = accept(server_fd, NULL, NULL);

char buf[1024];
int n = read(client_fd, buf, sizeof(buf));

write(client_fd, buf, n);

close(client_fd);
close(server_fd);
```

---

### 클라이언트

```c
int sock = socket(AF_INET, SOCK_STREAM, 0);

connect(sock, ...);

write(sock, "hello", 5);

char buf[1024];
int n = read(sock, buf, sizeof(buf));

close(sock);
```

---

## 6. TCP 연결 생명주기

### 연결 시작 (3-way handshake)

```
SYN → SYN+ACK → ACK
```

---

### 데이터 송수신

👉 바이트 스트림

---

### 연결 종료 (4-way handshake)

```
FIN → ACK → FIN → ACK
```

---

## 7. 핵심 요약

* 소켓 = 네트워크 입출력 인터페이스
* TCP = 바이트 스트림
* 세그먼트는 내부 구현일 뿐
* `accept()`는 새 소켓 생성
* send/recv는 **부분 처리 가능**
* 메시지 경계 없음 → 직접 처리해야 함

---

## 한 줄 정리

👉 **소켓 프로그래밍은 “스트림을 다루는 파일 I/O”라고 보면 된다**


## 1. 전체 흐름 (정확한 표현)

👉 **응용 → 소켓 → TCP → IP → 네트워크**

응용이 하는 일:

```c
write(sock, buf, len);
```

👉 이 순간 의미:

👉 **“바이트 스트림을 TCP에 넘긴다”**

---

## 2. TCP 입장에서 일어나는 일

응용이 준 데이터:

```
[aaaaabbbbbccccc...]
```

TCP는 이걸:

1. 내부 버퍼에 쌓음
2. 적당히 잘라서 (MSS 기준)
3. 세그먼트로 만들어 전송

---

## 3. 핵심 구분 (여기 중요)

| 계층      | 데이터 형태  |
| ------- | ------- |
| 응용 ↔ 소켓 | 바이트 스트림 |
| TCP 내부  | 세그먼트    |
| 네트워크    | 패킷      |

👉 그래서 말할 때 이렇게 해야 정확함:

👉 **“응용은 바이트 스트림을 TCP에 넘긴다” ⭕**
👉 “세그먼트를 보낸다” ❌ (응용 입장에서는 틀림)

---

## 4. 반대 방향 (수신)

네트워크 → TCP → 소켓 → 응용

TCP는:

1. 세그먼트 받음
2. 순서 맞춤
3. 이어붙임
4. 소켓으로 전달

응용은:

```c
read(sock, buf, len);
```

👉 그냥 **연속된 바이트를 읽을 뿐**

---

## 5. 한 줄로 완전히 정리

👉 **응용은 “바이트 스트림을 소켓에 쓰고”, TCP가 그걸 “세그먼트로 쪼개서 전송”한다**

---

## 6. 중요한 감각 하나

너가 보내는 이 코드:

```c
write(sock, "hello", 5);
write(sock, "world", 5);
```

TCP 입장:

👉 “helloworld” 하나의 스트림

---

## 최종 한 줄

👉 **소켓은 바이트 스트림 인터페이스이고, TCP는 그걸 쪼개서 보내는 엔진이다**
맞다. 정확히 그거다. 한 줄로 정리하면:

👉 **소켓 ↔ TCP 사이는 항상 “바이트 스트림”이다**

---

## 1. 방향별로 보면

### 응용 → TCP

```c
write(sock, buf, len);
```

👉 의미:
**“바이트 덩어리를 TCP에 넘김”**

---

### TCP → 응용

```c
read(sock, buf, len);
```

👉 의미:
**“연속된 바이트를 꺼내옴”**

---

## 2. 중요한 구분 (여기 핵심)

👉 이 구간에서는:

* 세그먼트 ❌
* 패킷 ❌
* 메시지 ❌

👉 오직:

👉 **바이트 스트림 ⭕**

---

## 3. 세그먼트는 어디서 나오냐

👉 TCP 내부 + 네트워크 구간에서만 존재

흐름:

```
응용
 ↓
소켓  ← 여기까지: 바이트 스트림
 ↓
TCP   ← 여기서: 세그먼트로 쪼갬
 ↓
IP    ← 패킷
```

---

## 4. 왜 이렇게 설계됐냐

이 구조 덕분에:

* 응용은 네트워크 신경 안 써도 됨
* MTU, 재전송, 순서 → 전부 TCP가 처리
* 응용은 그냥 read/write만 하면 됨

---

## 5. 한 번 더 핵심 정리

👉 **소켓 ↔ TCP: 바이트 스트림**
👉 **TCP 내부/네트워크: 세그먼트**


## 소켓은 구조체다

👉 **소켓 = “커널 내부에 있는 구조체(객체) + 그걸 가리키는 파일 디스크립터”**


### 1. 왜 “구조체”라고 보면 맞냐

네 감각 맞는 부분:

👉 커널 안에는 실제로 이런 형태로 존재함

* 상태 (LISTEN, ESTABLISHED)
* send buffer / recv buffer
* IP, PORT
* sequence number

👉 이걸 전부 담고 있는 게

👉 **커널의 소켓 구조체 (`struct socket`, `struct sock`)**

## 2. 근데 사용자 입장에서 중요한 차이

너가 코드에서 쓰는 건:

```c
int sock = socket(...);
```

👉 이건 구조체가 아니라:

👉 **파일 디스크립터 (fd)**

즉:

* 유저 공간 → 숫자만 들고 있음
* 커널 공간 → 진짜 구조체 존재

---

## 3. 관계 정리

```id="wz4z7m"
유저 프로그램
   ↓
파일 디스크립터 (int sock)
   ↓
커널 내부 소켓 구조체
   ↓
TCP/IP 스택
```

---

## 4. 핵심 포인트

👉 “소켓 = 구조체다”
→ **커널 내부 관점에서는 맞음 ⭕**

👉 “소켓 = 내가 쓰는 변수다”
→ **틀림 ❌ (그건 fd)**

---

## 5. 한 줄로 정확히

👉 **소켓은 커널에 있는 구조체고, 우리는 그걸 fd로 간접적으로 다룬다**

* 소켓 = 커널 객체 (구조체)
* fd = 그걸 가리키는 핸들
* 통신은 커널 TCP 스택이 수행

## 소켓내부
이건 한 줄로 정리된다:

👉 **“TCP 연결을 유지하고, 데이터 신뢰성 있게 보내기 위한 상태 정보”**

각 항목 역할을 정확히 쪼개서 보자.

---

### 1. 상태 (LISTEN, ESTABLISHED)

👉 **연결 단계 관리**

* `LISTEN` → 연결 기다리는 상태 (서버)
* `SYN_SENT` → 내가 연결 요청 보냄
* `ESTABLISHED` → 연결 완료 (데이터 송수신 가능)
* `FIN_WAIT`, `CLOSE_WAIT` → 종료 과정

👉 역할:

👉 **지금 이 소켓이 “무슨 단계인지” 판단해서 행동 결정**

예:

* LISTEN이면 데이터 못 보냄
* ESTABLISHED여야 send 가능

---

### 2. send buffer / recv buffer

#### send buffer

👉 **아직 네트워크로 다 안 나간 데이터 저장**

* `write()` 하면 여기 들어감
* TCP가 천천히 꺼내서 전송

👉 역할:

* 네트워크 느려도 앱 안 막힘
* 재전송 대비 (ACK 안 오면 다시 보냄)

---

#### recv buffer

👉 **네트워크에서 받은 데이터 임시 저장**

* TCP가 먼저 받아서 쌓아둠
* 앱이 `read()` 할 때 가져감

👉 역할:

* 앱이 느려도 데이터 안 날림
* out-of-order 재정렬 가능

---

### 3. IP, PORT

👉 **통신 상대 식별**

* 내 IP / 포트
* 상대 IP / 포트

👉 역할:

👉 **“이 데이터가 어느 소켓에 속하는지 구분”**

(핵심: 4-tuple)

```
(src IP, src PORT, dst IP, dst PORT)
```

---

### 4. sequence number

👉 **데이터 순서 + 유실 검출**

예:

```
1000~1499
1500~1999
```

👉 역할:

* 순서 맞춤
* 빠진 데이터 찾기
* 중복 제거

---

### 5. 전체 연결해서 보면

이 4개가 합쳐지면:

👉 TCP가 하는 4가지 핵심 기능 완성됨

| 기능    | 담당 요소           |
| ----- | --------------- |
| 연결 관리 | 상태              |
| 흐름 제어 | buffer          |
| 식별    | IP/PORT         |
| 신뢰성   | sequence number |

---

### 6. 한 줄 정리

👉 **이 정보들은 “TCP가 연결을 유지하면서, 순서 맞추고, 안 잃어버리고, 제대로 보내기 위한 최소 세트”다**

* 상태 → 연결 단계
* 버퍼 → 데이터 저장/흐름
* IP/PORT → 누구랑 통신
* sequence → 순서 + 재전송

---

여기까지 이해하면:

👉 “슬라이딩 윈도우 / congestion control”
👉 “왜 recv buffer 꽉 차면 속도 줄어드는지”
