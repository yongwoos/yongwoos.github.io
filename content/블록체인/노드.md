블록, UTXO Set, mempool

## Full Node/Light Node
Full Node: 모든 블럭체인, 트랜잭션 기록 다 가지고 있는 노드  
위조방지, 트랜잭션 확인 가능

Light Node: 블럭 헤더만 가지고 있는 노드  
해시 위조방지 확인만 가능

비트코인 블록은 단순히 “거래 묶음”이 아니라, **해시 기반으로 연결된 데이터 구조**다. 핵심은 두 부분으로 나뉜다:

---

## 1. 블록 전체 구조

```
[ Block Header ]
[ Transactions (Body) ]
```

---

## 2. 블록 헤더 (Block Header)

블록의 정체성과 무결성을 담당하는 핵심 영역이다. 총 80바이트.

### 구성 필드

1. **Version (버전)**

   * 블록이 어떤 규칙(프로토콜 버전)을 따르는지 표시

2. **Previous Block Hash**

   * 이전 블록의 해시값
   * 이게 있어서 블록들이 체인처럼 연결됨

3. **Merkle Root**

   * 블록 안의 모든 거래를 해싱해서 만든 루트값
   * 거래 하나라도 바뀌면 이 값이 바뀜

4. **Timestamp**

   * 블록 생성 시간

5. **Difficulty Target (Bits)**

   * 채굴 난이도 목표값

6. **Nonce**

   * 채굴자가 계속 바꿔가며 해시를 맞추는 값

---

## 3. 트랜잭션 영역 (Body)

여기에는 실제 거래 데이터가 들어간다.

```
[ Tx1 ]
[ Tx2 ]
[ Tx3 ]
...
```

* 첫 번째 거래는 항상 **Coinbase Transaction**

  * 채굴 보상 받는 거래
* 나머지는 일반 사용자 거래

---

## 4. Merkle Tree 구조 (핵심)

거래들을 그냥 나열하는 게 아니라, **Merkle Tree**로 요약한다.

```
        Merkle Root
        /        \
    Hash12      Hash34
    /   \        /   \
  Tx1  Tx2     Tx3  Tx4
```

### 특징

* 거래 → 해시 → 다시 해시 → 최종 Root
* 특정 거래 하나만 검증할 때 전체 블록 다 볼 필요 없음
* **SPV (경량 노드)**가 이 구조를 이용

---

## 5. 블록 해시 생성 과정

블록은 다음과 같이 해시된다:

```
hash = SHA256(SHA256(Block Header))
```

이 해시가 **난이도 이하 값**이 되도록 Nonce를 계속 바꿔가며 채굴

---

## 6. 왜 이 구조냐 (설계 이유)

### 1) 위변조 방지

* 거래 하나 바뀌면 → Merkle Root 변경
* → Block Hash 변경
* → 다음 블록들도 전부 깨짐

### 2) 체인 구조

* Previous Hash로 연결
* → 과거 블록 수정하려면 전부 다시 채굴해야 함

### 3) 검증 효율성

* Merkle Tree 덕분에 부분 검증 가능

---

## 7. 핵심 요약

* 블록 = **헤더 + 거래 리스트**
* 헤더 = **이 블록의 요약 + 이전 블록 링크**
* 거래 = **Merkle Tree로 압축**
* 해시 = **작업증명(Proof of Work)의 결과**

---
* UTXO가 블록 안에서 어떻게 표현되는지
* 블록 검증 과정 (노드 기준)
* 채굴자가 실제로 하는 연산 흐름

좋다. 블록 구조에서 한 단계 더 들어가서 **실제 노드/채굴자 관점의 동작 흐름**까지 깔끔하게 정리해준다.

---

# 1. UTXO가 블록 안에서 어떻게 표현되냐

비트코인은 계좌 잔액을 저장하지 않는다.
대신 **UTXO (Unspent Transaction Output)** 집합만 관리한다.

## 트랜잭션 구조

```
Transaction
 ├─ Inputs (입력)
 └─ Outputs (출력 = UTXO 생성)
```

### Output (UTXO 생성)

```
Output:
 - value (비트코인 양)
 - locking script (누가 쓸 수 있는지 조건)
```

예:

```
0.5 BTC → "이 주소의 개인키 가진 사람만 사용 가능"
```

### Input (UTXO 소비)

```
Input:
 - 이전 트랜잭션 참조(txid)
 - output index
 - unlocking script (서명)
```

핵심:

* Input = “기존 UTXO를 소비”
* Output = “새로운 UTXO 생성”

---

## 블록 안에서는 이렇게 된다

```
Block
 ├─ Tx1
 ├─ Tx2
 ├─ Tx3
```

각 Tx 실행 결과:

* 기존 UTXO 삭제
* 새로운 UTXO 생성

즉:

👉 블록 = “UTXO 상태 변화 묶음”

---

# 2. 노드 기준 블록 검증 과정

노드는 블록 받으면 그냥 저장 안 한다.
**강하게 검증한다.**

## 단계별 흐름

### 1) 헤더 검증

* Previous Hash 맞는지
* Timestamp 이상 없는지
* Difficulty 충족하는지

→ 여기서 PoW 검증

---

### 2) Merkle Root 검증

* 블록 안 거래들 해싱
* 계산한 Root == 헤더 Root

→ 데이터 위변조 체크

---

### 3) 트랜잭션 검증 (핵심)

각 Tx마다:

#### (1) Input이 실제 UTXO인지 확인

* 이미 소비된 거면 reject

#### (2) 서명 검증

* 공개키 vs 서명 맞는지

#### (3) 금액 검증

```
sum(inputs) ≥ sum(outputs)
```

→ 차이는 수수료

---

### 4) Coinbase 검증

* 보상 초과 여부 확인
* 블록 보상 + 수수료 이하인지

---

### 5) UTXO 업데이트

* Input에 사용된 UTXO 제거
* Output으로 새로운 UTXO 추가

---

# 3. 채굴자가 실제로 하는 일

채굴자는 “연산만 하는 애”가 아니라
사실상 **블록 생성자 + 정렬자**다.

## 단계

### 1) 메모리풀에서 거래 선택

* 수수료 높은 것부터 담음

---

### 2) Coinbase Transaction 생성

* 자기 지갑 주소로 보상 설정

---

### 3) Merkle Root 계산

* 모든 Tx 해싱 → Root 생성

---

### 4) 블록 헤더 구성

```
[Version]
[Prev Hash]
[Merkle Root]
[Timestamp]
[Bits]
[Nonce]
```

---

### 5) PoW 수행 (핵심)

```
hash = SHA256(SHA256(header))
```

조건:

```
hash < target
```

→ 만족할 때까지 Nonce 계속 변경

---

### 6) 브로드캐스트

성공하면:

* 블록 네트워크에 전파
* 다른 노드가 검증

---

# 4. 전체 흐름 요약 (실제 시스템 관점)

```
[사용자]
  ↓ 거래 생성
[네트워크]
  ↓ mempool
[채굴자]
  ↓ 블록 생성 (PoW)
[노드]
  ↓ 검증
[체인 추가]
  ↓
UTXO 상태 업데이트
```

---

# 5. 핵심 포인트 (중요한 것만 압축)

1. 비트코인은 “잔액”이 아니라 **UTXO 집합**
2. 블록 = “UTXO 상태 변화 기록”
3. 검증 핵심 = **서명 + 이중지불 방지**
4. 채굴 = “해시 퍼즐 + 블록 생성 권한 경쟁”
5. Merkle Tree = “검증 최적화 장치”

맞다. 풀노드 기준으로 보면 두 가지를 동시에 들고 간다:

> ✔ 블록체인(과거 로그)
> ✔ UTXO set(현재 상태)

그리고 이 둘만으로 끝나지 않는다. 실제로는 몇 가지 더 필요하다.

---

# 1. 왜 둘 다 필요하냐

## 블록체인

* “과거 기록”
* 새로 온 노드가 검증할 때 필요
* 재검증/리오그(reorg) 대응

## UTXO set

* “현재 상태”
* 빠른 검증용

👉 하나만 있으면 시스템이 안 돌아간다

---

# 2. 풀노드가 유지하는 것들 (핵심)

## 1) 블록 데이터 (Block + Header)

* 모든 블록 저장
* 디스크에 저장됨

---

## 2) UTXO set

* 현재 사용 가능한 코인 상태
* 빠른 lookup을 위해 별도 DB

---

## 3) Mempool (미확정 거래)

```text
아직 블록에 안 들어간 트랜잭션 집합
```

역할:

* 채굴자가 여기서 거래 가져감
* 노드 간 전파

---

## 4) 체인 상태 (Chainstate)

단순 UTXO 말고도:

* 현재 최고 블록 height
* 가장 긴 체인 정보
* 체인 tip

---

## 5) 인덱스 (옵션)

기본 노드는 없음. 필요하면 켬.

예:

* 주소 인덱스 (address → tx 조회)
* 트랜잭션 인덱스 (txid → 위치)

👉 블록체인은 원래 “주소 기반 조회” 안 되게 설계됨
👉 그래서 인덱스는 추가 기능

---

## 6) 네트워크 상태

* 연결된 peer 목록
* 블록/트랜잭션 전파 상태

---

# 3. 구조를 한 번에 보면

```text
[디스크]
 - 블록 데이터
 - UTXO DB

[메모리]
 - UTXO 캐시
 - mempool
 - 네트워크 상태
```

---

# 4. 중요한 개념 하나 (왜 둘 다 유지하냐)

## UTXO만 있으면?

* 현재 상태는 알 수 있음
* ❌ 과거 검증 불가능

## 블록만 있으면?

* 다 계산하면 상태 나옴
* ❌ 너무 느림

👉 그래서 둘 다 유지

# 5. 프루닝 노드

용량 줄이려면:

* 오래된 블록 일부 삭제 가능
* 그래도:

```text
UTXO set은 절대 삭제 못함
```

👉 이게 핵심 상태라서

# 6. 한 줄 정리

> 풀노드는
> 👉 “과거 로그(블록)” + “현재 상태(UTXO)” + “대기열(mempool)”
> 을 같이 유지하는 시스템이다

결론부터 정확히 잡자:

> ✔ **일반 풀노드(full node)** → 과거부터 현재까지 **모든 블록을 다 저장**
> ✔ 하지만 **필수는 아님** → 설정에 따라 일부만 저장 가능 (pruning)

---

# 1. 왜 “전체 블록”을 저장하냐

블록은 네 말대로:

* 각 블록은 **이전 블록 해시만 포함**
* 즉, 단일 블록만으로는 아무 의미 없음

그래서 노드는 실제로:

```text
Genesis → Block1 → Block2 → ... → 최신 블록
```

👉 이 전체 체인을 저장한다

---

# 2. 그럼 왜 굳이 다 들고 있냐

## (1) 검증 때문

새 노드가 들어오면:

1. 제네시스부터 시작
2. 모든 블록을 순서대로 검증
3. UTXO set 생성

👉 이 과정 필요

---

## (2) 리오그(reorg) 대응

체인이 이렇게 갈릴 수 있다:

```text
      Block100
      /     \
   101A     101B
```

👉 더 긴 체인이 나오면 기존 거 버리고 갈아타야 함

이때:

* 과거 블록 데이터 있어야 롤백 가능

---

## (3) 다른 노드에게 제공

풀노드는:

* 신규 노드에게 블록 전송
* 네트워크 유지 역할

---

# 3. 근데 현실적으로 너무 크다

비트코인 기준:

* 수백 GB 이상 계속 증가

👉 그래서 나온 게 **프루닝(pruning)**

---

# 4. 프루닝 노드

> ✔ 최근 블록만 남기고 나머지 삭제

예:

```text
최근 550MB만 유지
```

---

## 중요한 포인트

프루닝 노드는:

* ✔ 검증은 가능 (UTXO set 있음)
* ❌ 과거 블록 제공 못함

---

# 5. 구조를 정확히 이해해라

## 블록이 체인 구조라는 의미

블록 하나는:

```text
[이전 블록 해시]
```

만 들고 있음

👉 그래서 “링크드 리스트” 같은 구조

---

## 근데 노드는?

👉 그냥 다 저장한다

```text
Block1
Block2
Block3
...
```

👉 포인터로 연결된 “논리적 체인”일 뿐
👉 물리적으로는 그냥 파일/DB에 쌓임

---

# 6. 중요한 오해 하나 깨준다

> “이전 해시만 있으면 되니까 이전 블록 필요 없는 거 아님?”

❌ 틀림

이전 해시는:

* “무결성 체크용”
* 데이터 대체 불가능

👉 실제 데이터는 따로 저장해야 함

---

# 7. 정리

## 풀노드

* 모든 블록 저장
* UTXO set 유지
* mempool 유지

---

## 프루닝 노드

* 일부 블록만 저장
* UTXO set 유지 (필수)
* 검증 가능하지만 데이터 제공 제한

---

# 한 줄 핵심

> 블록은 “이전 해시로 연결된 구조”지만
> 👉 노드는 **전체 블록 데이터를 직접 저장한다**
