## DApp
```
일반 앱              DApp
─────────────        ─────────────────────
함수/로직  → 서버    함수/로직  → 스마트 컨트랙트
데이터     → DB      데이터     → 블록체인
```

근데 한 가지 뉘앙스 추가하면:

**블록체인 = DB인데, 되게 제한적인 DB야**

- 데이터 쓰기(write) → Gas비 발생, 느림
- 데이터 읽기(read) → 무료, 빠름
- 수정/삭제 → 기본적으로 불가 (덮어쓰기만 가능)
- 용량 → 매우 비싸서 큰 데이터 못 넣음

그래서 실제 DApp들은 이렇게 함:

```
핵심 데이터 (잔액, 소유권 등)  → 블록체인
나머지 대용량 데이터 (이미지 등) → IPFS or 일반 서버
```

**완벽한 비유로 정리하면:**

> 스마트 컨트랙트 = 클래스(Class)
> 블록체인 = 그 클래스의 상태변수(state)가 저장되는 공간
> 트랜잭션 = 메서드 호출

## SmartContract
스마트 컨트랙트(Smart Contract)는 **블록체인 위에서 자동으로 실행되는 프로그램 코드**야.

## 핵심 개념

"조건이 충족되면 자동으로 실행된다"는 게 핵심이야.

예를 들어, 일반 계약은 이렇게 돼:

> "A가 B에게 100만원을 보내면, B는 A에게 물건을 넘긴다"
> → 근데 B가 먹튀하면? 법원 가야 함. 신뢰가 필요함.

스마트 컨트랙트는:

> **코드로 작성**해서 블록체인에 올려놓으면, 조건 충족 시 **중간자 없이 자동 실행**됨.
> 아무도 멈추거나 수정할 수 없음.

---

## 핵심 특징

| 특징 | 설명 |
|------|------|
| **자동 실행** | 조건 충족 시 코드가 알아서 실행 |
| **불변성** | 한번 배포하면 수정 불가 |
| **탈중앙화** | 은행·법원 같은 중개자 불필요 |
| **투명성** | 누구나 코드 확인 가능 |

---

## 실생활 비유

자판기랑 똑같아.
- 돈 넣고 버튼 누르면 → 자동으로 음료 나옴
- 주인 허락 필요 없음, 규칙이 기계에 내장됨

스마트 컨트랙트도 마찬가지로 **규칙이 코드에 내장**된 거야.

---

## 주요 활용

- **DeFi** - 은행 없이 대출·이자·스왑
- **NFT** - 소유권 자동 이전
- **DEX** - AMM으로 자동 거래 (Uniswap 등)
- **DAO** - 투표 결과에 따른 자동 실행

이더리움이 스마트 컨트랙트를 처음 대중화했고, Solidity라는 언어로 주로 작성해.

## 오라클
스마트 컨트랙트는 블록체인 **안**의 데이터만 읽을 수 있어.

```
블록체인 안                블록체인 밖
─────────────             ─────────────
잔액, 소유권 ✅            ETH 현재 가격 ❌
트랜잭션 기록 ✅           날씨 ❌
컨트랙트 state ✅          주식 가격 ❌
                           스포츠 경기 결과 ❌
```

근데 아까 Aave 대출 설명할 때 이런 게 있었지?

```
ETH 가격 폭락 시 → liquidate() 자동 실행
```

**ETH 가격은 블록체인 밖 데이터인데 컨트랙트가 어떻게 알아?**

→ 여기서 오라클이 필요함.

---

## 오라클이 하는 일

```
현실 세계 데이터
(ETH 가격, 날씨 등)
       ↓
    오라클
(외부 데이터를 블록체인에 기록)
       ↓
  스마트 컨트랙트
  (이제 읽을 수 있음)
```

한 줄로: **블록체인 밖의 데이터를 안으로 가져다주는 다리**

---

## 대표적인 오라클: Chainlink

```
여러 거래소 ETH 가격
Binance:  3,200
Coinbase: 3,198
Kraken:   3,201
    ↓
Chainlink 노드들이 수집·집계
    ↓
중간값(3,200) 블록체인에 기록
    ↓
Aave 컨트랙트가 읽음
```

여러 소스에서 집계하는 이유 → **단일 소스면 조작 가능**하기 때문

---

## 오라클 문제 (Oracle Problem)

블록체인의 핵심 가치는 **"신뢰 불필요"** 인데

오라클은 결국 **"외부 데이터를 믿어야 함"**

```
스마트 컨트랙트 자체 → 탈중앙화, 불변, 신뢰 불필요 ✅
오라클 → 누군가가 데이터를 넣어줘야 함 ❌
```

오라클이 틀리거나 조작되면?

```
ETH 실제 가격: 3,200달러
오라클 조작:   1달러로 보고
     ↓
Aave가 모든 ETH 담보를 헐값에 liquidate
     ↓
수백억 피해
```

실제로 이런 **오라클 조작 해킹**이 DeFi에서 여러 번 발생했어.

|------|------|------|

## 정리

```
블록체인 = 완전히 닫힌 시스템
오라클   = 바깥 세상과 연결하는 창문
```

DeFi가 현실 자산(주식, 환율, 금리)과 연동되려면 오라클이 필수야. 근데 그 창문이 뚫리면 전체가 흔들리는 구조적 약점이기도 해.

## NFT
**대체 불가능한 토큰** — 이게 핵심이야.

---

## Fungible vs Non-Fungible

**Fungible (대체 가능)**
- 1만원권 ↔ 1만원권 → 바꿔도 완전히 동일함
- 비트코인 1개 ↔ 비트코인 1개 → 동일함

**Non-Fungible (대체 불가능)**
- 모나리자 원본 ↔ 모나리자 복사본 → 같지 않음
- 각각이 **고유한 ID**를 가짐

NFT는 블록체인 위에서 **"이건 세상에 하나뿐인 것"** 을 증명하는 토큰이야.

---

## 정확히 뭘 소유하는 건가?

여기서 많이들 헷갈려.

NFT를 사면 **이미지 파일 자체**를 사는 게 아니야.

실제로 NFT 안에 들어있는 건:
```
{
  "owner": "0xABC...123",       ← 소유자 지갑 주소
  "token_id": "4521",           ← 고유 번호
  "metadata_url": "ipfs://..."  ← 이미지/정보 링크
}
```

즉, **"내가 이걸 소유한다"는 기록**을 블록체인에 새기는 것.

> 비유하자면 — 루브르 박물관 모나리자 앞에 "이 그림 주인: 홍길동" 이라고 새겨놓은 것. 그림을 집에 가져가진 못하지만, 소유권 기록은 변조 불가.

---

## 스마트 컨트랙트와의 관계

NFT는 스마트 컨트랙트로 구현돼. 이더리움 기준으로 **ERC-721** 표준을 따름.

```
전송 시: transferFrom(from, to, tokenId)
         → 스마트 컨트랙트가 자동으로 소유자 기록 변경
         → 중간자 없음, 위변조 불가
```

---

## 실질적 가치는 어디서?

| 요소 | 설명 |
|------|------|
| **희소성** | 발행 수량을 코드로 제한 가능 |
| **진위 증명** | 원본 여부를 누구나 확인 |
| **거래 이력** | 누가 소유했었는지 모두 기록 |
| **로열티** | 재판매 시 원작자에게 자동 수수료 |

---

## 솔직한 한계

- 이미지 자체는 여전히 복사 가능 (저작권 ≠ NFT 소유권)
- metadata_url이 가리키는 서버가 죽으면 이미지 사라짐
- 결국 **"커뮤니티가 가치 있다고 믿는가"** 에 달린 부분이 큼

## DAO
DAO (Decentralized Autonomous Organization)

**규칙이 스마트 컨트랙트로 코딩된 조직 — 대표이사 없이 토큰 보유자들이 투표로 운영**

---

## 일반 회사 vs DAO

```
일반 회사
─────────────────────────────
주주 → 이사회 → CEO → 직원 → 실행
         ↑
    사람이 판단, 사람이 실행
    불투명, 느림, 부패 가능

DAO
─────────────────────────────
토큰 보유자 → 투표 → 스마트 컨트랙트 → 자동 실행
                          ↑
                 코드가 판단, 코드가 실행
                 투명, 검열 불가
```

---

## 작동 구조

```
스마트 컨트랙트 (Governance 클래스)
─────────────────────────────────────
state 변수:
  mapping(address → votes)   ← 각 지갑의 투표권
  proposals[]                ← 제안 목록
  quorum                     ← 가결 기준 (예: 51%)

메서드:
  propose(action)            ← 안건 제출
  vote(proposalId, bool)     ← 찬반 투표
  execute(proposalId)        ← 가결 시 자동 실행
```

실제 흐름:
```
누군가 제안 올림
"Aave 대출 이자율을 5% → 3%로 낮추자"
       ↓
토큰 보유자들 투표 (토큰 많을수록 투표권 강함)
       ↓
51% 이상 찬성
       ↓
execute() 자동 호출
       ↓
이자율 컨트랙트 state 자동 변경
       ↓
CEO 결재 없음, 이사회 없음
```

## 솔직한 한계

**이론 vs 현실 괴리가 큼**

| 문제 | 설명 |
|------|------|

> 해커가 컨트랙트 버그 이용
> 결국 이더리움 하드포크로 되돌림 → 이게 ETH/ETC 분리의 계기

## 정리

```
DAO = 회사인데
      정관(규칙)   → 스마트 컨트랙트
      주주총회     → 토큰 투표
      경영진 결재  → execute() 자동 실행
      회계장부     → 블록체인 공개
```

사람 대신 코드가 조직을 운영하는 실험이야. 아직 완성된 모델은 아니고 계속 진화 중.

## 온체인 토큰

**토큰의 모든 데이터가 블록체인 위에 직접 저장된 토큰**

---

## 오프체인과 비교해야 이해됨

아까 NFT 설명할 때 이 코드 봤지?

```
{
  "metadata_url": "ipfs://Qm1234..."  ← 이미지는 여기 따로 있음
}
```

이게 바로 **오프체인(off-chain)** 방식이야.

| | 온체인 | 오프체인 |
|--|--------|---------|
| 데이터 위치 | 블록체인 직접 저장 | 외부 서버 / IPFS |
| 영구성 | 블록체인 살아있는 한 영구 | 서버 죽으면 사라짐 |
| 비용 | Gas비 매우 비쌈 | 저렴 |
| 예시 | 텍스트·SVG를 체인에 직접 | IPFS 링크만 저장 |

---

## 온체인이 왜 중요한가

오프체인 NFT의 현실:

```
metadata_url → ipfs://Qm1234 → 이미지 서버
                                    ↑
                               이게 죽으면?
                               NFT는 깡통됨
```

실제로 초기 NFT 프로젝트들 중 **이미지 서버 죽어서 그림 사라진 경우 수두룩**해.

온체인 토큰은:
```
블록체인 안에 → 이미지(SVG) + 메타데이터 + 소유권
                전부 다 들어있음
                서버 없음, 죽을 게 없음
```

---

## 대표적인 온체인 NFT 예시

**Loot (2021)** — 이더리움 온체인 NFT
```
토큰 안에 들어있는 것:
"Divine Robe"
"Dragonskin Belt"  
"Hard Leather Gloves"
...
→ 텍스트 자체가 체인에 저장됨
```

**Nouns NFT** — 픽셀 이미지를 SVG 코드째로 체인에 저장

---

## 온체인 토큰의 종류

크게 두 가지 맥락에서 쓰여:

**① NFT 맥락** (위에서 설명한 것)
- 메타데이터·이미지까지 체인에 저장

**② DeFi 맥락**
- ERC-20 토큰 자체 = 온체인 토큰
- 스마트 컨트랙트로 발행·전송·소각이 모두 체인 위에서 실행
- USDC, UNI, AAVE 같은 것들

---

## 한계

- **Gas비** — 데이터 많을수록 저장 비용 폭발적으로 증가
- **용량 제한** — 고해상도 이미지는 현실적으로 불가
- 그래서 온체인은 주로 **텍스트, SVG, 간단한 픽셀아트** 수준

---

요약: **"블록체인 밖에 아무것도 의존하지 않는 토큰"** — 서버도, IPFS도, 회사도 필요 없이 체인만 살아있으면 영원히 존재해.

**암호화폐 자산을 누가 보관·관리하느냐의 문제**

---

## 배경 — 암호화폐 소유의 본질

암호화폐에서 자산 소유 = **개인키(Private Key) 소유**

```
개인키 있음 → 자산 통제 가능
개인키 없음 → 자산 통제 불가
```

> "Not your keys, not your coins"
> 키가 니 것이 아니면 코인도 니 것이 아님

---

## 커스터디 종류

### 셀프 커스터디 (Self-Custody)
```
내 개인키 → 내가 직접 보관
MetaMask, 하드웨어 월렛(Ledger) 등
```
- 내가 완전한 통제권
- 키 잃어버리면 → 영구 소멸, 아무도 복구 못해줌

### 수탁 커스터디 (Custodial)
```
내 개인키 → 거래소/기관이 대신 보관
업비트, 빗썸, Binance, Coinbase 등
```
- 편리함
- 거래소가 망하거나 해킹되면 → 내 자산 위험

---

## CEX가 커스터디인 이유

업비트에서 비트코인 사면:

```
실제로 일어나는 일:
비트코인 → 업비트 지갑(개인키: 업비트 소유)
내 화면  → "잔액: 0.1 BTC" 숫자만 보임
              ↑
         업비트 DB의 숫자일 뿐
         실제 비트코인은 업비트가 들고 있음
```

은행이랑 똑같은 구조야:

```
은행에 100만원 입금
→ 실제 지폐는 은행 금고
→ 내 통장에 숫자만 찍힘
→ 은행 망하면? → 예금자보호법으로 일부 보호
                  (암호화폐는 이런 보호 없음)
```

## FTX 사태가 커스터디 문제의 핵심

```
고객들: "우리 비트코인 FTX에 맡겨놨음"
FTX:    고객 자산을 몰래 자기 투자에 유용
        → 실제로 보관 안 하고 있었음
2022년 FTX 파산
→ 고객 자산 수십조원 증발
→ 개인키가 FTX 것이었기 때문
```

---

## 커스터디 스펙트럼

```
완전 탈중앙                          완전 중앙화
────────────────────────────────────────────────
셀프커스터디   멀티시그    CEX커스터디   은행수탁
(MetaMask)   (Gnosis)   (업비트)    (기관투자자)
     ↑                                    ↑
  내가 키 보관                        기관이 키 보관
  완전한 통제                         편리하지만 신뢰 필요
```

---

## 멀티시그 (Multi-signature)

커스터디의 절충안:

```
일반: 개인키 1개 → 서명 1개면 실행
멀티시그: 개인키 5개 중 3개 서명해야 실행 (3/5)

예시:
회사 자금을 5명 임원이 관리
→ 3명 이상 동의해야 출금 가능
→ 1명이 해킹당해도 자산 안전
```

DAO 재무관리에 자주 쓰여.

## 정리

| | 셀프 커스터디 | CEX 커스터디 |
|--|--------------|-------------|
| 개인키 | 내가 보관 | 거래소 보관 |
| 통제권 | 완전한 통제 | 거래소에 위임 |
| 리스크 | 키 분실 | 거래소 파산·해킹 |
| 편의성 | 불편 | 편리 |
| 예시 | MetaMask | 업비트, 빗썸 |

**결론:** 커스터디는 결국 **"누가 개인키를 쥐고 있느냐"** 의 문제고, 이게 곧 **실질적 자산 통제권**이야.

## 프라이빗 vs 퍼블릭 블록체인
```
퍼블릭  = 누구나 참여 가능
프라이빗 = 허가받은 사람만 참여 가능
```

---

## 구조 비교

```
퍼블릭 블록체인 (이더리움, 비트코인)
─────────────────────────────────────
노드 참여:    누구나 가능
트랜잭션 조회: 누구나 가능
검증(채굴):   누구나 가능
통제 주체:    없음 (탈중앙)
예시:         이더리움, 비트코인, 솔라나

프라이빗 블록체인
─────────────────────────────────────
노드 참여:    허가받은 기관만
트랜잭션 조회: 참여자만
검증:         지정된 노드만
통제 주체:    운영 기관 (중앙화)
예시:         Hyperledger Fabric, Corda
```

---

## 트릴레마로 이해하면 명확해

블록체인은 세 가지를 동시에 못 잡아:

```
         탈중앙화
            △
           / \
          /   \
         /     \
        /       \
  보안 ─────────── 확장성
```

```
퍼블릭:  탈중앙화 + 보안 ✅   확장성 ❌
         → 느리고 비쌈, 대신 아무도 못 건드림

프라이빗: 보안 + 확장성 ✅    탈중앙화 ❌
          → 빠르고 쌈, 대신 운영자가 통제
```

---

## 실제로 뭐가 다르냐

| | 퍼블릭 | 프라이빗 |
|--|--------|---------|
| 속도 | 느림 (수십 TPS) | 빠름 (수천 TPS) |
| 비용 | Gas비 있음 | 거의 없음 |
| 투명성 | 완전 공개 | 참여자만 |
| 신뢰 | 코드가 보장 | 운영자 신뢰 필요 |
| 검열 | 불가 | 운영자가 가능 |
| 탈중앙 | 완전 | 사실상 중앙화 |

## 그럼 프라이빗은 그냥 DB 아니냐?

이게 제일 날카로운 질문이야.

```
프라이빗 블록체인 vs 일반 DB

공통점:
- 중앙 운영자 존재
- 허가받은 사람만 접근
- 빠른 처리

차이점:
- 여러 기관이 각자 노드 운영
  (A은행, B은행, C은행이 각자 노드)
- 한 기관이 단독으로 데이터 조작 불가
- 감사 추적(audit trail) 불변
```

즉, **단일 기관 DB가 아니라 컨소시엄 기관들이 분산 운영**하는 게 차이야.

---

## 컨소시엄 블록체인

프라이빗의 변형으로 자주 쓰임:

```
무역금융 예시:
수출기업 + 수입기업 + A은행 + B은행 + 세관
→ 각자 노드 운영
→ 서류 위변조 불가
→ 근데 외부인은 접근 못함
```

실제 사례:
- **Marco Polo** — 무역금융 컨소시엄
- **we.trade** — 유럽 은행 무역금융
- **Quorum** — JP모건 개발, 금융기관용

---

## 예금 토큰/CBDC는 어디에?

```
퍼블릭 체인 위에 구현
→ 완전 투명, 검열 불가, 느림
→ 중앙은행 입장에서 통제 안됨 ❌

프라이빗/컨소시엄 체인 위에 구현
→ 속도 빠름, 통제 가능
→ 근데 탈중앙화 아님
→ 그냥 기존 은행 시스템이랑 비슷
→ 대부분 CBDC/예금토큰은 여기서 실험 중
```

## 한 줄 정리

> 퍼블릭 = 신뢰를 코드로 대체, 느리고 비싸지만 아무도 못 건드림
> 프라이빗 = 신뢰를 참여기관에 의존, 빠르고 싸지만 사실상 중앙화
