title: UTXO

BTC는 **계좌 잔액** 개념이 없고, **UTXO(Unspent Transaction Output)** 모델을 사용

## UTXO란?

거래가 발생하면 "아직 쓰지 않은 출력값"들이 생기는데, 그게 UTXO야.

예를 들어:
- 누군가 나한테 0.3 BTC 보냄 → UTXO ①
- 또 누군가 나한테 0.5 BTC 보냄 → UTXO ②

이 두 개가 내 지갑 주소에 연결된 UTXO야.

## 잔액 = UTXO 합산

"내 잔액"은 **내 주소에 연결된 모든 UTXO의 합**이야.

```
잔액 = UTXO① (0.3) + UTXO② (0.5) = 0.8 BTC
```

별도로 "잔액 필드"가 블록체인에 저장되는 게 아니라, 노드가 **UTXO 셋 전체를 추적**하면서 그때그때 합산하는 방식.

## 왜 이 방식을 씀?

- **이중지불 방지**가 쉬움 → UTXO는 한 번 쓰면 소멸, 새로운 UTXO 생성
- **병렬 검증** 가능 → 서로 다른 UTXO를 쓰는 트랜잭션은 독립적으로 검증 가능
- **프라이버시** → 주소마다 여러 UTXO가 흩어져 있음

## ETH와 비교

| | BTC | ETH |
|---|---|---|
| 모델 | UTXO | Account/Balance |
| 잔액 | UTXO 합산 | 계좌에 직접 저장 |
| 구조 | 복잡하지만 검증에 유리 | 직관적 |

ETH는 은행 계좌처럼 잔액이 직접 상태에 저장돼서 단순하지만, BTC의 UTXO 모델이 보안·검증 측면에선 더 견고해.

## 신규 노드 동기화 과정 (IBD: Initial Block Download)

### 1단계: 피어 연결
```
새 노드 → DNS seed 서버에 물어봄
         → 기존 노드 IP 목록 받음
         → 여러 피어에 연결
```

### 2단계: 블록 다운로드
- 피어들한테 블록 요청
- 제네시스 블록(블록 #0)부터 최신까지 순서대로 받음
- 여러 피어한테 **병렬로** 받아서 속도 높임

### 3단계: 검증 + UTXO 셋 재구성
```
블록 #0 검증 → UTXO 셋 업데이트
블록 #1 검증 → UTXO 셋 업데이트
블록 #2 검증 → UTXO 셋 업데이트
...
블록 #90만+ 검증 → 완료
```
모든 블록을 처음부터 **직접 재계산**해서 UTXO 셋 만듦.

---

### 신뢰는?

- 피어를 **신뢰하지 않음**
- 받은 블록을 전부 **직접 검증** (작업증명, 서명, 이중지불 등)
- 과반수 채굴자가 악의적이지 않은 한 가장 긴 체인 = 유효한 체인

---

### 얼마나 걸림?

Bitcoin Core 기준 현재:
- 블록 데이터: ~**600GB+**
- 시간: 하드웨어에 따라 **수 시간 ~ 수 일**
- assumevalid 옵션 쓰면 오래된 블록 서명검증 스킵해서 빠르게 가능 (기본값으로 켜져 있음)

응 원칙적으로는 그게 맞는데, 현실적으로는 최적화가 있어.

## 원칙: 전체 재계산

제네시스부터 최신 블록까지 전부 replay → UTXO 셋 구성
→ 시간 엄청 걸림

---

## 현실: assumeUTXO (최적화)

Bitcoin Core 0.24부터 도입된 기능이야.

```
기존 방식:
블록 0 → 1 → 2 → ... → 90만 → UTXO 셋 완성 (수일)

assumeUTXO:
특정 높이의 UTXO 셋 스냅샷을 하드코딩된 해시로 검증
→ 일단 그걸 쓰고 바로 사용 가능
→ 백그라운드에서 전체 검증은 계속 진행
```

### 신뢰 문제는?
- 스냅샷 해시가 **Bitcoin Core 소스코드에 하드코딩**되어 있음
- 즉 개발자들이 검증한 값 → 그걸 믿는 거
- 백그라운드 full 검증이 끝나면 완전한 trustless 상태

---

## 정리

| 방식 | 속도 | 신뢰 |
|---|---|---|
| 전체 재계산 | 느림 (수일) | 완전 trustless |
| assumeUTXO | 빠름 (즉시 사용) | 개발자 신뢰 필요 (일시적) |

사실 대부분의 일반 사용자는 풀노드 안 띄우고 **라이트 노드(SPV)** 쓰는데, 그건 UTXO 셋 자체를 안 갖고 피어한테 물어보는 방식이야.

응, 근데 그건 **자동**으로 돼. 초기 동기화랑은 다른 얘기야.

## 초기 동기화 완료 후

노드가 최신 블록까지 따라잡으면, 그 이후엔:

```
새 블록 채굴됨
    ↓
피어들한테 "나 새 블록 있어" 알림 (inv 메시지)
    ↓
내 노드가 "줘봐" 요청
    ↓
블록 받아서 검증
    ↓
UTXO 셋 자동 업데이트
    ↓
또 다른 피어들한테 전파
```

**실시간으로 계속 듣고 있는 거야.** 노드가 켜져 있는 한.

---

## 노드 껐다 켰을 때는?

```
노드 재시작
    ↓
피어 재연결
    ↓
꺼져 있던 동안 생긴 블록들만 요청해서 받음
    ↓
UTXO 셋 업데이트
```

껐던 시간만큼만 따라잡으면 되니까 금방 돼.

---

핵심은 노드가 **항상 피어랑 연결 유지**하면서 새 블록을 push 방식으로 받는 구조라서, 별도로 "동기화 해야지" 하고 수동으로 할 필요가 없어.

## 예시
## 트랜잭션 UTXO 예시

### 상황
A가 B한테 0.3 BTC 보내려 함. A는 0.5 BTC짜리 UTXO 하나 갖고 있음.

---

### 트랜잭션 구조

```
[ Input ]
  UTXO: A의 0.5 BTC (이전 트랜잭션의 output)
  → 이걸 통째로 소비

[ Output ]
  → B 주소: 0.3 BTC   (새 UTXO 생성)
  → A 주소: 0.19 BTC  (잔돈, 새 UTXO 생성)
  → 채굴자: 0.01 BTC  (수수료, 명시 안 해도 자동 계산)
```

---

### UTXO 셋 변화

```
트랜잭션 전:
  A의 UTXO: 0.5 BTC ✓

트랜잭션 후:
  A의 UTXO: 0.5 BTC ✗ (소멸)
  B의 UTXO: 0.3 BTC ✓ (생성)
  A의 UTXO: 0.19 BTC ✓ (잔돈으로 생성)
```

---

### 포인트

- Input은 반드시 **기존 UTXO 전체를 소비**해야 함 (일부만 쓰는 개념 없음)
- 잔돈은 **자기 주소로 다시 Output** 만들어야 함
- Input 합계 - Output 합계 = **채굴자 수수료** (자동으로 채굴자가 가져감)
- 잔돈 Output 안 만들면? → **전부 수수료로 날아감** (실수 사례 실제로 있음)
